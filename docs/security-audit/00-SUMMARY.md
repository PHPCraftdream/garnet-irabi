## Топ-находки, требующие внимания (по убыванию важности)

### 1. `POST /dev-login` — выдача admin/owner-сессии и снос БД, единственная защита — эвристика по файлам на диске
Отчёты: `03-auth-middlewares.md` (F-1, F-2, F-5), `04-services-entities-db.md` (F1)
Эндпоинт `Foreground/Controllers/DevLoginController.php` (`post__main`, `post__resetDb`) не требует пароля/OTP/CSRF и защищён только `Env::isDevDir()` — проверкой, есть ли в 6 родительских каталогах папка `.idea`/`.vscode`/`.vs`/`.xcodeproj`/`.atom`. Это чисто серверная проверка (не зависит от запроса), поэтому **в корректно собранном prod она не эксплуатируема**. Но если такая директория случайно попадёт в prod-артефакт (например, через `.idea` в git-чекауте или CI), эндпоинт превращается в:
- мгновенный вход под ролью `admin`/`owner`/`moderator` без пароля (`role=admin`),
- `TRUNCATE` таблиц `balance_ledger`, `bookings`, `payments`, `time_slots` и т.д. + удаление тестовых аккаунтов (`post__resetDb`).
**Рекомендация:** добавить явный позитивный флаг (`env=dev` из `app.ini`), требуемый ВМЕСТЕ с `isDevDir()` (как это уже сделано в `WorkerScopeMiddleware::isDevContext`), либо вообще не регистрировать роут вне dev-сборки; исключить IDE-директории из deploy-артефакта.

### 2. Модератор может начислять/списывать баланс любому аккаунту без ограничений (High)
Отчёт: `02-dashboard-controllers.md` (H-1)
`DashboardFinanceController::post__adjustBalance` защищён только `isModerator()` (низший staff-уровень), без верхней границы суммы и без запрета целиться в аккаунты owner/admin. Позволяет самому себе или кому угодно начислить произвольную сумму.
**Рекомендация:** поднять требование до `isOwner()`, добавить лимит суммы и запрет на цель с равным/более высоким рангом.

### 3. Модератор может заблокировать (`IS_DISABLED`) аккаунт owner/admin — вертикальная эскалация (High)
Отчёт: `02-dashboard-controllers.md` (H-2)
`DashboardUsersController::post__setUserFlag` не проверяет ранг цели относительно вызывающего. Модератор может отключить владельца/админа, потенциально закрепившись у власти.
**Рекомендация:** перед применением `IS_DISABLED`/`IS_APPROVED`/`type` проверять, что цель не выше по рангу вызывающего; запретить самоцель.

### 4. Начисление платежей неодобренному/отключённому эксперту в обход UI-гейта (Medium)
Отчёт: `01-foreground-controllers.md` (находки 1–2)
Публичный список экспертов фильтрует неодобренных, но сама транзакция бронирования (`SlotsController::post__book`, `BookingsController::post__book`) не проверяет `is_approved`/`IS_DISABLED` владельца слота — только его прямую ссылку/id. Неодобренный эксперт может получать деньги, минуя ручное одобрение модератором.

### 5. Модератор имеет полный доступ ко всей финансовой/персональной информации (Medium)
Отчёт: `02-dashboard-controllers.md` (M-1), и возможна утечка тел писем в mail-log (M-2) — нужно перепроверить, действительно ли `body_html` попадает в ответ для не-admin (агент отметил это как требующее верификации в адаптере фреймворка, не смог подтвердить однозначно).

### 6. Баланс/леджер: нет транзакции и защиты от отрицательного баланса на уровне примитива (Medium)
Отчёт: `04-services-entities-db.md` (F2)
`FwBalanceLedger::addEntry` + `FwAccountBalance::recalculate` не оборачиваются в транзакцию, нет `SELECT … FOR UPDATE`, нет идемпотентности по `ref_type/ref_id`. Риск двойного списания при гонке/реплее запроса — но конкретная достижимость зависит от контроллеров списания (не входило в файловый скоуп этого прохода, нужен точечный follow-up по `BalanceController`/`BookingsController`).

### Прочее (Low, детали в файлах)
- `/sys/log` — публичный, без CSRF, без rate-limit эндпоинт записи в лог-файлы (мягкий DoS диска, log-spam) — `01-foreground-controllers.md` №5, `03-auth-middlewares.md` F-3.
- `POST /users/~preview`, `GET /profile`, `expert/~userPreview` отдают настоящее имя (а в одном месте — email) отключённых пользователей вместо анонимизированной заглушки, которая используется везде остальные — `01-foreground-controllers.md` №3.
- `POST /im/~send` не проверяет бизнес-границу получателей (обычный пользователь может писать любому другому обычному пользователю, а не только экспертам/модераторам) — `01-foreground-controllers.md` №4.
- Раздача `text/plain`/`text/log` вложений inline без `X-Content-Type-Options: nosniff` — `05-uploads-external-im.md` F-05-01.
- Admin-only загрузка OG/CMS-картинок (`PublicImageUploadTrait`) не переэнкодит изображение через GD (в отличие от аватаров) и подменяет запрещённое расширение на `.jpg` вместо отказа — `05-uploads-external-im.md` F-05-02.

## Что проверено и НЕ является уязвимостью

- **SQL-инъекции** — не найдены нигде (все пользовательские значения идут через bind-параметры; интерполируются только имена таблиц из конфига/констант).
- **CSRF** — на всех state-changing POST в Foreground и Dashboard включена глобальная проверка `Origin` + `CSRF_TOKEN` (`hash_equals`, constant-time).
- **RCE через загрузку файлов** (`.php`/`.phtml`/`.htaccess`) — заблокирован двойным whitelisting (расширение + real-MIME) и server-generated именами файлов.
- **Path traversal** — везде используется `basename()` + `realpath()`-containment.
- **IDOR в бронированиях, слотах, IM, тикетах поддержки** — везде явные проверки владения (`account_id === session account`).
- **XSS (stored/reflected)** — не найдено ни в Twig (`|raw` не используется), ни в React (единственный `dangerouslySetInnerHTML` рендерит статичную i18n-строку, не пользовательский ввод).
- **SSRF** — серверных fetch пользовательских URL нигде нет; `ExternalController` — чисто interstitial-страница.
- **Mass assignment роли/баланса через self-service профиль** — поля `type`/`IS_ADMIN`/`IS_APPROVED` и т.п. исключены из whitelist `editFields()`.
- **`SysOpcacheResetController`** — корректный fail-closed токен с `hash_equals`.
- **`.test` авто-логин / `TestScope`** — требует секрет на диске (`.allow_tests`) + constant-time сравнение заголовка; не флиппается запросом; проходит через тот же CSRF/Origin гейт.

## Итоговая приоритизация действий

1. Закрыть `/dev-login` вторым позитивным флагом (`env=dev`) + исключить IDE-маркеры из деплоя — самое высокое последствие при отказе защиты (полный захват + снос БД).
2. Поднять `post__adjustBalance` до `isOwner()` и добавить лимит суммы.
3. Добавить проверку ранга цели в `post__setUserFlag`/`post__setUserType`.
4. Добавить проверку `is_approved`/`IS_DISABLED` эксперта в транзакции бронирования, а не только в публичном списке.
5. Решить продуктово вопрос видимости финансов/PII модератором (M-1/M-2) и обернуть леджер в транзакцию с защитой от отрицательного баланса (F2) — по согласованию с архитектурой.
6. Остальное — низкоприоритетный hardening-бэклог (см. Low-находки в файлах 01, 03, 05).
