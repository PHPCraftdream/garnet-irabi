## Session summary

This (very long, multi-day) session took `phpcraftdream/garnet-framework` from "not ready for Packagist" to a published, tested, security-hardened package, and used the real-world app `garnet-irabi` as the integration testbed throughout.

**Framework release track:** Started from an external review listing 4 blockers (no git tags, app:create hardcoding a path-repo forever, broken Quickstart, hardcoded CLI version) plus secondary items (extensions, CI, SECURITY.md, CONTRIBUTING.md). Fixed all of them, then iteratively tagged `v0.1.0-alpha1` through `v0.1.0-alpha10` on `PHPCraftdream/garnet-framework`, publishing each to Packagist via a GitHub Actions workflow (`.github/workflows/packagist.yml`, uses `PACKAGIST_TOKEN` secret + `PACKAGIST_USERNAME` var) and re-verifying end-to-end after every fix using real installs (via `composer require`, via a purpose-built `Installer/bin/garnet-new` one-shot installer, and via the real app `IRabi`). Bugs found and fixed only by actually installing as a Packagist dependency (not just in the monorepo): `bin/garnet` autoload fallback path, `garnet serve` binding to `0.0.0.0` instead of `localhost` + not printing full URL, `WorkDir/Config/` template shipping empty (fresh apps 500'd immediately), `base_url` port mismatch (8002 vs actual 8001), `tooling/mcp/` and `tooling/server/` wrongly excluded from the Packagist dist archive (`.gitattributes`), `Admin/dist/` prebuilt assets excluded by a blanket `.gitignore` rule, `GarnetDeployDiffCommand` hardcoding monorepo `Framework/`/`Apps/<App>/` path prefixes (broken for vendor-installed apps — now uses `GarnetRunner::$frameworkDir`/`$appDir` as two real anchors), `Env::isDevDir()` climbing directories from its own `__DIR__` instead of the real app root (broke dev-only UI like the `AuthDev` panel for any vendor install), sticky-footer CSS gap, `FwAccountBalance::recalculate()` lost-update race (fixed with a single atomic `INSERT…SELECT…ON DUPLICATE KEY UPDATE`), and `FwBalanceLedger::addEntry()` missing idempotency (fixed with a `UNIQUE INDEX` on `(account_id, entry_type, ref_type, ref_id)` + duplicate-key no-op). Old alpha1–9 tags were deleted from GitHub (user request) once alpha10 was confirmed good, and Packagist was re-synced — only alpha10 remains as a tagged release now (`dev-master` still tracks HEAD).

**IRabi track:** Converted `Apps/IRabi` from the monorepo path-repo/`@dev` layout to a real Packagist dependency (`^0.1.0-alpha4` → bumped alongside every framework alpha, now alpha10), verified against a real native MySQL57 Windows service (not Docker — user preference). Both `garnet-framework` and `garnet-irabi` GitHub repos are now **public**. Dual-licensed MIT/Apache-2.0 (matching the framework) with a real author contact email (`phpcraftdream@gmail.com`) added to both `composer.json` files. IRabi's own CI (`.github/workflows/ci.yml`) was completely rewritten — the old one assumed a pre-Packagist monorepo-sibling-checkout-plus-jq-patch scheme that silently no longer applied; the new one just does a normal `composer install` against Packagist, and deliberately **excludes e2e from CI** (user's explicit ask, to avoid loading GitHub Actions) — e2e stays a local/on-demand `composer test:e2e` command.

## Decisions

- **Deleted old Packagist alpha tags (v0.1.0-alpha1–9) from GitHub** rather than leaving them, once alpha10 was confirmed working — user explicitly wanted a clean version list; accepted as a one-time exception to "avoid destructive git ops" since it's pre-1.0 alpha churn on a package with no known external consumers yet.
- **Chose atomic single-statement SQL (`INSERT…SELECT…ON DUPLICATE KEY UPDATE` / `UNIQUE INDEX` + duplicate-key catch) over explicit transactions+row locks** for both the balance-cache race and the ledger idempotency gap — simpler, no lock contention window, matches a pattern already used elsewhere in the codebase.
- **Rejected Docker for local MySQL** — user wants the native Windows `MySQL57` service used throughout instead; agents were explicitly redirected mid-task when they defaulted to `docker run`.
- **Kept `.vscode/` as a git-tracked dev marker in IRabi** rather than treating it as a leak risk, given `DevLoginController` now requires the additional `$globals->isDev()` request-based gate (real prod under nginx+php-fpm never satisfies it) — but still added IDE-marker names to `GarnetBundleCommand`'s copy-exclude list as cheap defense-in-depth per user's explicit call.
- **Declined to build a separately-published `phpcraftdream/garnet-installer` Packagist package** (would need a subtree-split to a second repo) — kept `Installer/` as an in-monorepo convenience tool only, since `composer global require phpcraftdream/garnet-framework` already covers the same use case once `bin/garnet`'s autoload bug was fixed.
- **Did not touch `docs/customer-handover.md`'s NO-GO verdict or its acceptance checklist** even though the underlying blockers it lists are now fixed — flagged as an open follow-up rather than silently rewriting a document that reads like a formal sign-off artifact.
