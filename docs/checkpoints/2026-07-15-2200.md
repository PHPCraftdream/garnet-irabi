This turn's work was on **report 12** (`docs/security-audit/12-ms-postfix-authorization-review.md`), which found 4 issues, all now confirmed real and fixed in the working tree (not yet committed):
- **M-01**: `/expert/id~N` (ExpertController::get__main) checked only `expert_profiles.is_approved`, which doesn't get cascade-cleared when a moderator disables/demotes an account — a disabled or demoted expert's profile (name partially anonymized) still leaked their future free slot ids and stats. Fixed by gating on `UserEntityConfig::isApprovedActiveExpert($expertId)` (the same account-level predicate — type=expert, account IS_APPROVED, not IS_DISABLED — already used by the booking path) instead of the bare `expert_profiles` row.
- **M-02**: `ImController::canMessage()` allowed messaging any account with an `expert_profiles` row regardless of account-level type/approval/disabled state, for both the sender-is-expert and recipient-is-expert branches. Fixed by replacing both `ExpertProfiles::selectOneByField()` truthy checks with `UserEntityConfig::isApprovedActiveExpert()`. The "existing conversation always allowed" bypass was deliberately left untouched.
- **M-03**: `/user/id~N` (UserProfileController) exposed any authenticated actor's name + booking/cancellation counters for ANY other regular user, no restriction, no disabled anonymization. User was asked to pick a policy and chose: visible to self / moderator+ / an expert who has actually had a booking from that user (real counterparty) — everyone else gets 404. Implemented as a new private `canViewProfile()` helper querying TimeSlots+Bookings for the counterparty case.
- **L-01** (Low): `CommentsController::post__create()` validated the comment target only by `expert_profiles` row existence, not account-level active/approved. Fixed identically via `isApprovedActiveExpert()`.

User also asked mid-task why these checks aren't middleware — I explained the established pattern in this codebase: actor-based checks (role/rank of the person making the request) are middleware because they're uniform per route group; these findings are all target-based checks (business entity referenced by a request parameter that varies per-endpoint: URL id vs POST recipient_id vs POST entity_id), which don't fit a single router-level middleware and are instead expressed as an explicit reusable helper (`UserEntityConfig::isApprovedActiveExpert()`, already established from the booking-path fix in an earlier report) called inline by each controller.

Also worth knowing for continuity: this session repeatedly hit two recurring environment gotchas, now well understood:
1. Editing a PHP file requires restarting the ENTIRE `garnet serve` process tree (kill all `php -S` worker processes AND the node wrapper, not just a `php garnet cache:opcache` CLI call) before a Playwright run will see the change — opcache `revalidate_freq=2` normally would auto-refresh, but the actual observed behavior needed full worker-pool restart to reliably pick up edits.
2. A genuine (dev-server-only, not production-relevant) framework bug was discovered: `Account::$sessionAccount` (Account.php:55/68 in garnet-framework) is a process-level static that doesn't verify it matches the CURRENT request's session before returning — the shared `php -S` worker pool used by `garnet serve` can serve a stale cached identity across two different concurrently-live browser sessions that happen to land on the same long-lived worker process. This caused flaky/wrong results in the A-01 owner-vs-admin test until restructured to do one login→test→context.close() at a time instead of holding two staff sessions open concurrently. Not fixed (out of scope, doesn't affect php-fpm production), just documented and worked around in test structure.
3. `test.describe.configure()` must be called INSIDE a `test.describe(...)` callback, never at file top level — doing it at top level produces a cryptic "Playwright Test did not expect test.describe() to be called here" error that looks like environment flakiness but is actually a real code mistake. Hit this multiple times this session.
4. Never use shell `&` for backgrounding Playwright runs — two concurrent `npx playwright test` invocations against the same isolation template corrupt each other's migration/seed state (confirmed by direct observation this session). Always use the Bash tool's `run_in_background` parameter instead, and never run two isolation-based Playwright invocations concurrently.
